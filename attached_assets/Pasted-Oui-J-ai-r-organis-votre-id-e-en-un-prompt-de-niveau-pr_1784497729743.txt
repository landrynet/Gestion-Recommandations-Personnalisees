Oui. J'ai réorganisé votre idée en un **prompt de niveau professionnel**, en tenant compte de l'architecture existante. Le but n'est pas de reconstruire le système, mais de **le faire évoluer sans casser les données** ni les fonctionnalités déjà développées.

---

# Phase 3 : Évolution de la gestion des années scolaires, migration des données et archivage

Nous entamons une nouvelle phase du développement du système. L'objectif est de mettre en place une gestion complète des années scolaires, de la migration des données et de l'archivage, tout en garantissant la cohérence, la sécurité et l'évolutivité de l'application.

**Important :** Le système est déjà en production de développement. Les nouvelles fonctionnalités doivent être intégrées sans casser l'architecture actuelle, sans perte de données et sans modifier les fonctionnalités existantes.

## 1. Intégration de la notion de Niveau

Actuellement, le système gère les **Sections** et les **Classes**. Les classes sont déjà liées aux sections.

Afin de préparer la gestion des promotions et des nouvelles années scolaires, il est désormais nécessaire d'ajouter une nouvelle entité : **Niveau**.

La nouvelle hiérarchie devient :

**Année scolaire → Niveau → Section → Classe**

Exemple :

* Niveau : 7ème

  * Section : Scientifique

    * Classe : A
    * Classe : B
* Niveau : 8ème

  * Section : Scientifique

    * Classe : A
    * Classe : B

Chaque classe devra être liée à un niveau et à une section.

Cette évolution doit être réalisée **sans supprimer ni casser les données existantes**. Une migration de la base de données devra être prévue afin de rattacher les classes actuelles au niveau correspondant.

Le niveau devra contenir au minimum :

* Nom
* Ordre (pour la promotion automatique)
* Cycle (si nécessaire dans le futur)

---

# 2. Gestion des années scolaires

Une seule année scolaire peut être active.

Toutes les données du système doivent être rattachées à une année scolaire :

* élèves
* enseignants
* affectations
* notes
* bulletins
* absences
* résultats
* publications
* communications
* portail parent

Les anciennes années doivent rester consultables mais ne plus être modifiables.

---

# 3. Clôture d'une année scolaire

Avant l'ouverture d'une nouvelle année, le préfet lance un assistant de clôture.

Le système vérifie automatiquement :

* toutes les périodes sont terminées ;
* les bulletins sont générés ;
* les résultats sont validés ;
* les publications sont effectuées ;
* aucune donnée importante n'est incomplète.

Si une anomalie est détectée, la clôture est bloquée jusqu'à correction.

---

# 4. Archivage

Lorsqu'une année est clôturée :

Toutes les données deviennent **historiques**.

Elles restent consultables mais ne peuvent plus être modifiées.

Les éléments archivés comprennent notamment :

* bulletins ;
* résultats ;
* notes ;
* publications ;
* statistiques ;
* rapports ;
* affectations ;
* absences.

Le portail parent devra disposer d'un espace **Historique / Archives** permettant de consulter les anciennes années scolaires.

Les résultats publiés d'une année clôturée devront apparaître automatiquement dans cette rubrique.

---

# 5. Création d'une nouvelle année scolaire

Le préfet crée une nouvelle année scolaire.

Le système ouvre ensuite un assistant de préparation.

---

# 6. Migration intelligente des données

Lors de la création d'une nouvelle année, le système demande quelles informations doivent être reprises.

Les éléments suivants devront être migrés automatiquement :

### Structure pédagogique

* niveaux
* sections
* classes
* matières
* coefficients
* périodes
* paramètres pédagogiques

### Utilisateurs

* préfet
* enseignants
* administrateurs

### Paramètres

* identité de l'établissement
* logo
* paramètres PWA
* couleurs
* paramètres généraux

### Modèles

Les modèles déjà configurés devront également être conservés.

Notamment :

* modèle officiel de bulletin
* paramètres d'impression
* modèles de documents
* modèles de publication

Ces modèles ne devront jamais être recréés chaque année.

Ils font partie de la configuration permanente de l'établissement.

---

# 7. Données qui ne doivent jamais être copiées

Les éléments suivants doivent rester propres à chaque année scolaire :

* notes
* moyennes
* classements
* bulletins générés
* résultats
* absences
* publications
* journaux d'activités

La nouvelle année doit démarrer avec des données vierges.

---

# 8. Migration des affectations

Le système demande :

Souhaitez-vous reprendre les affectations des enseignants ?

Si oui :

Toutes les affectations sont copiées.

Le préfet peut ensuite les modifier.

Cette fonctionnalité évite de refaire toute l'organisation chaque année.

---

# 9. Promotion des élèves

Le système analyse automatiquement les décisions finales.

Selon les résultats :

* Admis → promotion au niveau supérieur
* Redouble → maintien dans le même niveau
* Transféré → non réinscrit
* Diplômé / Fin de cycle → archivage

Le système propose automatiquement la nouvelle classe en fonction du niveau suivant.

Le préfet peut modifier chaque proposition avant validation.

---

# 10. Conservation de l'historique

Chaque élève doit conserver tout son parcours scolaire.

Exemple :

2024–2025 → 7ème Scientifique A

2025–2026 → 8ème Scientifique B

2026–2027 → 1ère Scientifique A

Toutes les années restent consultables.

---

# 11. Sécurité de la migration

La migration doit être entièrement transactionnelle.

Si une erreur survient :

* aucune donnée ne doit être partiellement copiée ;
* le système doit revenir automatiquement à l'état précédent ;
* aucune incohérence ne doit apparaître.

---

# 12. Journal des opérations

Toutes les opérations doivent être enregistrées :

* création d'une année ;
* clôture ;
* migration ;
* promotion des élèves ;
* activation ;
* archivage.

Avec :

* utilisateur responsable ;
* date ;
* heure.

---

# 13. Vérification finale

Avant l'activation de la nouvelle année, le système affiche un résumé complet :

* année créée ;
* nombre de niveaux ;
* sections ;
* classes ;
* enseignants ;
* élèves promus ;
* redoublants ;
* diplômés ;
* transférés ;
* données migrées.

Le préfet valide définitivement avant activation.

---

# 14. Contraintes techniques

Cette évolution doit respecter les principes suivants :

* ne casser aucune fonctionnalité existante ;
* préserver toutes les données actuelles ;
* effectuer les migrations de base de données proprement ;
* éviter les duplications de données ;
* conserver une architecture claire et évolutive ;
* optimiser les performances ;
* respecter les bonnes pratiques Django et PostgreSQL ;
* garantir la sécurité, l'intégrité et la cohérence des données.

Toutes les nouvelles fonctionnalités devront être compatibles avec les futures évolutions du système (gestion avancée des promotions, statistiques, portail parent, historiques, rapports et nouvelles fonctionnalités), afin d'éviter toute refonte de la base de données à l'avenir.

---

## Objectif final

Cette évolution doit transformer le système en une **plateforme scolaire professionnelle**, capable de gérer plusieurs années scolaires de manière fiable. Chaque année devra être totalement indépendante tout en conservant un historique complet des données. La migration devra être intelligente, sécurisée et sans perte d'information. L'ajout du **Niveau** devra s'intégrer naturellement à l'architecture existante (Sections et Classes), afin de faciliter les promotions automatiques, les changements de structure et les évolutions futures, tout en garantissant la stabilité et la cohérence de l'ensemble du système.

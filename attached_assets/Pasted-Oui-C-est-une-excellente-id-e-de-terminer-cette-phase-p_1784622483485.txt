Oui. C'est une excellente idée de terminer cette phase par un **audit final**, puis seulement si tout est validé, de générer automatiquement un **jeu de données réalistes** pour les tests. Voici un prompt que vous pouvez donner à Codex.

---

# AUDIT FINAL, VALIDATION COMPLÈTE ET GÉNÉRATION DES DONNÉES DE TEST

Nous arrivons à la fin de la phase de stabilisation.

Avant toute nouvelle fonctionnalité, effectuer un audit complet du projet afin de vérifier que le système est entièrement fonctionnel, cohérent et prêt pour une campagne de tests avec des données réalistes.

Aucune nouvelle fonctionnalité ne doit être ajoutée durant cette étape.

---

# 1. Audit complet

Analyser l'ensemble du projet.

Vérifier notamment :

* toutes les pages accessibles ;
* tous les menus ;
* toutes les routes ;
* tous les formulaires ;
* tous les boutons ;
* tous les liens ;
* toutes les API ;
* tous les traitements métier ;
* toutes les permissions ;
* tous les rôles ;
* toutes les redirections ;
* toutes les validations de formulaires ;
* tous les calculs ;
* toutes les impressions ;
* tous les exports ;
* toutes les notifications.

Identifier :

* les erreurs ;
* les incohérences ;
* les doublons ;
* les composants inutilisés ;
* les imports inutiles ;
* le code mort ;
* les risques de sécurité ;
* les problèmes de performance.

Corriger automatiquement tout ce qui peut l'être sans modifier le comportement fonctionnel attendu.

---

# 2. Vérification fonctionnelle

Contrôler que chaque module fonctionne correctement :

* authentification ;
* première connexion ;
* changement de mot de passe ;
* gestion des utilisateurs ;
* gestion des élèves ;
* gestion des enseignants ;
* classes ;
* sections ;
* cours ;
* maxima ;
* évaluations ;
* encodage des notes ;
* calcul des résultats ;
* génération des bulletins ;
* publication des résultats ;
* portail parent ;
* QR Code ;
* Progressive Web App ;
* paramètres de l'établissement ;
* tableaux de bord.

Aucune erreur ne doit subsister.

---

# 3. Vérification des données

Contrôler que :

* aucune donnée n'est perdue lors des modifications ;
* toutes les relations entre les tables sont cohérentes ;
* aucune clé étrangère n'est cassée ;
* aucune suppression ne crée de données orphelines ;
* toutes les contraintes sont respectées.

---

# 4. Vérification des performances

Vérifier :

* les requêtes SQL ;
* les requêtes répétées ;
* les traitements inutiles ;
* les temps de chargement ;
* les composants trop lourds ;
* les éventuelles fuites mémoire.

Optimiser lorsque cela est possible.

---

# 5. Vérification de la sécurité

Contrôler :

* les permissions ;
* les rôles ;
* les accès directs par URL ;
* les protections CSRF ;
* la validation des entrées utilisateur ;
* les injections SQL ;
* les attaques XSS ;
* les téléversements de fichiers ;
* les mots de passe ;
* les sessions.

Corriger toute faille détectée.

---

# 6. Validation finale

Lorsque tous les contrôles sont terminés, produire un rapport indiquant :

* les problèmes détectés ;
* les corrections appliquées ;
* les optimisations réalisées ;
* les éventuels points restant à surveiller.

Si des erreurs bloquantes subsistent, **ne pas passer à l'étape suivante** tant qu'elles ne sont pas corrigées.

---

# 7. Génération d'un jeu de données réalistes

**Uniquement si toutes les vérifications précédentes sont validées**, créer un script de peuplement de la base de données (`seed`) permettant de générer un environnement de test complet.

Le script doit être **idempotent** (exécutable plusieurs fois sans créer de doublons), utiliser les modèles Django, respecter les contraintes et les relations de la base de données, et produire des données cohérentes.

Le jeu de données devra comprendre notamment :

* 1 établissement scolaire ;
* 1 préfet ;
* 20 enseignants ;
* 2 à 5 sections ;
* plusieurs options si nécessaires ;
* 15 classes réparties par niveau ;
* 30 à 40 élèves par classe (environ 450 à 600 élèves) ;
* des parents associés aux élèves ;
* les utilisateurs et comptes correspondants ;
* des cours affectés aux enseignants ;
* des maxima variés ;
* plusieurs périodes scolaires ;
* des évaluations ;
* des notes réalistes (répartition crédible, sans valeurs aléatoires incohérentes) ;
* les moyennes calculées ;
* les classements ;
* les bulletins générés ;
* les résultats publiés ;
* les QR Codes des élèves ;
* les données nécessaires au portail parent.

Toutes les données doivent respecter les règles métier existantes.

---

# 8. Plan de tests

Après la génération des données, produire un plan de tests fonctionnels détaillé couvrant au minimum :

* connexion et déconnexion ;
* gestion des utilisateurs ;
* gestion des élèves ;
* encodage des notes ;
* calcul des résultats ;
* génération et publication des bulletins ;
* consultation du portail parent ;
* accès par QR Code ;
* accès manuel au portail parent ;
* installation et fonctionnement de la PWA ;
* contrôle des permissions selon les rôles ;
* tests sur mobile, tablette et ordinateur.

Le plan devra préciser les scénarios, les résultats attendus et les critères de réussite.

---

# Objectif final

À la fin de cette étape, le projet devra être considéré comme **prêt pour une recette fonctionnelle complète**, avec une base de données réaliste, un script de génération reproductible, un rapport d'audit, un plan de tests détaillé et une application stable, performante, sécurisée et prête à être utilisée dans un environnement scolaire réel.

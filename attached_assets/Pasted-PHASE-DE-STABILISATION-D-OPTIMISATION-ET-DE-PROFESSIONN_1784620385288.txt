PHASE DE STABILISATION, D'OPTIMISATION ET DE PROFESSIONNALISATION DU SYSTÈME

Le système est désormais suffisamment avancé. À partir de cette étape, nous suspendons temporairement le développement de nouvelles fonctionnalités.

L'objectif est de stabiliser, corriger, optimiser et harmoniser toutes les fonctionnalités existantes afin d'obtenir une application fiable, cohérente, sécurisée et prête pour des tests complets puis un déploiement en établissement scolaire.

Toutes les modifications doivent être réalisées sans casser les fonctionnalités existantes.

Avant toute modification, analyser le projet afin d'identifier les incohérences, les traitements dupliqués, les fichiers inutilisés, les composants obsolètes et les optimisations possibles.

1. Amélioration de la gestion des élèves

Lorsqu'un utilisateur modifie les informations d'un élève :

tous les champs doivent être préremplis avec les valeurs déjà enregistrées ;
la date de naissance ne doit jamais être perdue ;
elle doit rester affichée automatiquement et ne changer que si l'utilisateur la modifie volontairement.

La photo de profil de l'élève doit être utilisée partout dans le système :

profil de l'élève ;
résultats ;
listes ;
portail parent ;
toute interface affichant un élève.

Si aucune photo n'existe, afficher une image par défaut.

2. Gestion des utilisateurs

Le préfet est responsable de la gestion des comptes des enseignants.

Il peut :

créer un utilisateur ;
modifier un utilisateur ;
supprimer un utilisateur ;
réinitialiser son mot de passe.

Lors de la création :

le système génère automatiquement un mot de passe temporaire sécurisé ;
l'adresse e-mail devient l'identifiant officiel de connexion.

Lors de la première connexion :

l'utilisateur se connecte avec son e-mail et son mot de passe temporaire ;
il est immédiatement obligé de définir un nouveau mot de passe fort avant d'accéder au système.

Le mot de passe temporaire devient définitivement invalide après cette opération.

Le mot de passe doit respecter une politique de sécurité :

longueur minimale configurable ;
majuscule ;
minuscule ;
chiffre ;
caractère spécial.

Afficher un indicateur de robustesse du mot de passe.

Le compte du préfet suit exactement le même fonctionnement lors de sa première connexion.

Préparer également l'architecture pour une future authentification à deux facteurs (2FA), sans l'implémenter pour l'instant.

3. Réinitialisation du mot de passe

Le comportement actuel présente un dysfonctionnement.

Lorsque le bouton Réinitialiser le mot de passe est utilisé, la page devient floue et inutilisable.

Corriger ce problème.

La réinitialisation doit être intégrée directement dans la page de modification de l'utilisateur.

Depuis cette page, le préfet doit pouvoir :

modifier les informations ;
réinitialiser le mot de passe ;
supprimer le compte si nécessaire.

L'interface doit rester fluide.

4. Bulletins scolaires

Je fournirai un modèle officiel de bulletin.

Adapter entièrement le bulletin existant à ce modèle.

Le bulletin devient la source officielle des résultats.

Chaque période doit afficher :

les notes ;
les totaux ;
le pourcentage ;
la place de l'élève.

Lorsque les informations existent, afficher également :

la moyenne générale ;
la place générale.

Le portail parent ne doit jamais recalculer les résultats.

Toutes les données affichées doivent provenir du bulletin.

5. Portail parent

Le portail parent devient uniquement une interface de consultation.

Il ne doit jamais recalculer :

les moyennes ;
les pourcentages ;
les classements.

Il doit uniquement afficher les données officiellement publiées.

Les périodes non publiées doivent rester invisibles.

6. Accès au portail parent

Le fonctionnement actuel du QR Code est satisfaisant et doit être conservé.

Le QR Code ouvre directement le portail parent.

La PWA installée depuis ce portail doit toujours ouvrir directement le portail parent.

Aucune redirection vers la connexion du personnel ne doit être effectuée.

Ajouter deux modes d'accès :

Accès par QR Code

Ajouter un bouton Scanner un QR Code.

Lorsqu'il est utilisé :

la caméra s'ouvre directement ;
le parent peut scanner immédiatement.
Accès manuel

Le parent peut également accéder au portail en renseignant :

le nom complet de l'élève ;
le mot de passe fourni par l'établissement ;
l'établissement scolaire (si le système est multi-établissements).

Les deux méthodes doivent donner exactement le même résultat.

Une fois le parent authentifié, mémoriser son accès de manière sécurisée afin qu'il n'ait pas à ressaisir ses informations à chaque ouverture de la PWA. Prévoir également une option permettant de changer d'élève ou de scanner un nouveau QR Code.

7. Progressive Web App (PWA)

Revoir complètement la configuration de la PWA.

Vérifier :

manifest.json ;
start_url ;
icônes ;
favicon ;
splash screen ;
métadonnées ;
redirections.

Le logo officiel de l'établissement doit être utilisé partout.

Le nom, le nom court, la description, les couleurs et les icônes doivent être configurables.

8. Identité de l'établissement

Créer une section :

Paramètres → Identité de l'établissement

Le préfet pourra configurer :

logo officiel ;
nom de l'établissement ;
nom complet de la PWA du système ;
nom court ;
description ;
nom complet de la PWA du portail parent ;
nom court ;
description ;
couleur principale ;
couleur d'arrière-plan.

Toutes ces informations doivent être utilisées automatiquement dans :

la PWA ;
les bulletins ;
le portail parent ;
les pages de connexion ;
les favicons ;
les icônes.

Le système doit pouvoir être utilisé par plusieurs établissements sans modifier le code.

9. Gestion des maxima

Créer une gestion complète des maxima.

Le préfet peut :

ajouter un maximum ;
modifier un maximum ;
supprimer un maximum.

Les maxima doivent être automatiquement triés du plus petit au plus grand.

Exemple :

10 → 15 → 20 → 30 → 40 → 50 → 60 → 80 → 100

Toutes les listes utilisant les maxima doivent respecter automatiquement cet ordre.

10. Publication des résultats

Simplifier complètement le processus.

Éviter les étapes intermédiaires inutiles.

Le processus doit être logique :

vérification ;
publication ;
génération du bulletin ;
disponibilité immédiate dans le portail parent.
11. Interface utilisateur

Uniformiser :

les boutons ;
les couleurs ;
les icônes ;
les tableaux ;
les formulaires ;
les cartes ;
les messages ;
les animations.

Toutes les pages doivent avoir la même identité visuelle.

12. Performances

Optimiser le système.

Réduire :

les requêtes inutiles ;
les calculs redondants ;
les traitements en double.

Les indicateurs de chargement doivent fonctionner uniquement pendant les traitements réels.

Aucun délai artificiel ne doit être utilisé.

13. Optimisation du code

Effectuer un audit complet du projet.

Identifier et supprimer :

les fichiers inutilisés ;
les composants inutilisés ;
les imports inutiles ;
les dépendances inutilisées ;
le code mort ;
les duplications.

Réorganiser le projet afin d'obtenir une architecture propre, lisible et évolutive.

14. Sécurité

Vérifier :

les permissions ;
les rôles ;
les accès ;
les redirections.

Chaque utilisateur ne doit accéder qu'aux fonctionnalités correspondant à son rôle.

Les mots de passe doivent être stockés avec le mécanisme de hachage sécurisé de Django.

15. Contrôle qualité

Avant de considérer cette phase terminée, vérifier que :

aucune erreur JavaScript n'apparaît dans la console ;
aucune erreur serveur n'est générée ;
les permissions sont correctes ;
les performances sont optimisées ;
le responsive fonctionne sur mobile, tablette et ordinateur ;
les bulletins, le portail parent et les tableaux de bord affichent exactement les mêmes données ;
toutes les fonctionnalités existantes continuent de fonctionner correctement.
Objectif final

Cette phase a pour objectif de transformer le système en une application stable, cohérente, performante, sécurisée et professionnelle. La priorité n'est plus d'ajouter de nouvelles fonctionnalités, mais de perfectionner celles qui existent déjà, d'éliminer les incohérences, de simplifier les parcours utilisateurs, d'optimiser le code et les performances, et de garantir une expérience fiable pour les préfets, les enseignants, les parents et les futurs établissements qui utiliseront la plateforme.

Important : À la fin de cette phase, le système devra être prêt pour une campagne complète de tests fonctionnels avec des données réelles, afin de valider chaque module avant de reprendre le développement des nouvelles fonctionnalités.

MISE À JOUR DU PROJET – CORRECTION DU PORTAIL PARENTS, OPTIMISATION DES PERFORMANCES ET AMÉLIORATION DE LA PWA
Contexte

L'application est déjà entièrement développée et fonctionnelle.

Les modules existants sont :

Back Office (Préfet des études)
Espace Enseignant
Gestion des élèves
Gestion des enseignants
Gestion des classes
Gestion des matières
Gestion des notes
Modèles de bulletins
Génération des bulletins
Publication des résultats
Portail Parents
Cartes d'élèves avec QR Code
Progressive Web App (PWA)

Avant toute modification :

analyser complètement l'architecture du projet ;
comprendre la logique métier existante ;
ne supprimer aucune fonctionnalité ;
respecter le design actuel ;
préserver la stabilité de l'application.
1. Correction critique de l'erreur HTTP 500

Le portail Parents fonctionne correctement jusqu'à l'authentification.

Le problème apparaît uniquement lorsqu'un parent tente d'afficher un bulletin dont les résultats ont été publiés.

Scénario
Le Préfet publie les résultats.
Le parent ouvre le portail.
Le parent s'authentifie correctement.
Le système doit afficher le bulletin.
À ce moment, une erreur HTTP 500 (Internal Server Error) est renvoyée et le bulletin ne s'affiche pas.

Je ne souhaite pas une correction basée sur des suppositions.

Je souhaite une analyse complète de la cause réelle.

Tu dois :

reproduire le problème ;
consulter les logs Django et la stack trace sur Replit ;
identifier précisément la cause ;
corriger le problème à la source ;
vérifier que le problème est définitivement résolu.

Analyser notamment :

les vues Django ;
les modèles ;
les templates ;
les URLs ;
les sessions ;
les permissions ;
les relations entre les modèles ;
la récupération de la publication active ;
la récupération des notes ;
les calculs (totaux, pourcentages, rangs) ;
les requêtes SQLite ;
les variables envoyées aux templates.

Ne jamais masquer l'erreur avec un try/except générique.

Corriger uniquement la cause réelle.

Si une donnée est absente, afficher un message clair à l'utilisateur au lieu d'une erreur technique.

2. Vérification complète

Tester les scénarios suivants :

création d'un élève ;
génération du QR Code ;
génération de la carte ;
activation du portail ;
création du code d'accès ;
publication des résultats ;
consultation du bulletin ;
actualisation de la page ;
nouvelle consultation ;
réinitialisation de l'accès ;
nouvelle activation.

Aucune étape ne doit provoquer une erreur HTTP 500.

3. Optimisation des performances et expérience utilisateur (PWA)

L'application est déjà une Progressive Web App.

Je souhaite maintenant améliorer considérablement ses performances.

Chargement intelligent

Je ne souhaite aucun délai artificiel.

Ne jamais utiliser :

setTimeout()
temporisation fixe
attente simulée

Le Loader doit fonctionner uniquement à partir des événements réels de l'application.

Il doit :

apparaître dès qu'une requête commence ;
rester visible pendant le traitement réel ;
disparaître uniquement lorsque la réponse est reçue et que la page est complètement prête.

Le système doit détecter automatiquement :

les requêtes HTTP ;
les requêtes AJAX ou Fetch ;
les soumissions de formulaires ;
les changements de page ;
les appels Django ;
les réponses du serveur.
Barre de progression

Ajouter une barre de progression moderne en haut de l'écran (type NProgress ou équivalent).

Fonctionnement :

démarrer automatiquement au début d'une navigation ou d'une requête ;
progresser pendant le traitement réel ;
disparaître automatiquement lorsque la page est totalement chargée.

Ne jamais utiliser de délai fixe.

Boutons intelligents

Lorsqu'un utilisateur clique sur un bouton :

désactiver immédiatement le bouton ;
afficher un spinner dans le bouton ;
empêcher les doubles clics ;
réactiver le bouton uniquement lorsque l'opération est réellement terminée.

Appliquer ce comportement à toutes les opérations importantes :

Connexion
Enregistrement
Publication
Recherche
Consultation
Impression
Génération des bulletins
4. Optimisation du système

Analyser entièrement le projet afin d'améliorer les performances.

Optimiser :

les requêtes SQLite ;
les requêtes Django ORM (select_related, prefetch_related lorsque c'est pertinent) ;
les vues ;
les calculs répétitifs ;
les traitements inutiles ;
le chargement des images ;
les fichiers CSS ;
les fichiers JavaScript ;
le cache des ressources statiques de la PWA.

Ne jamais modifier la logique métier.

L'objectif est uniquement d'accélérer l'application.

5. Navigation fluide

La navigation doit être plus réactive.

Réduire :

les rechargements inutiles ;
les requêtes dupliquées ;
les appels répétitifs vers la base de données.

L'application doit donner immédiatement un retour visuel à l'utilisateur lorsqu'une action est lancée.

6. Analyse des performances

Avant toute optimisation :

analyser les performances actuelles.

Après optimisation :

mesurer les performances obtenues ;
comparer les résultats avant/après ;
expliquer les optimisations réalisées.
7. Rapport final

À la fin des travaux, fournir un rapport détaillé comprenant :

la cause exacte de l'erreur HTTP 500 ;
les fichiers modifiés ;
les corrections apportées ;
les optimisations réalisées ;
les composants ajoutés (loader, barre de progression, boutons intelligents) ;
les tests effectués ;
les recommandations pour améliorer encore la stabilité et les performances.
Contraintes
Ne supprimer aucune fonctionnalité existante.
Respecter totalement l'architecture actuelle.
Conserver le design actuel.
Écrire un code propre, modulaire et documenté.
Tester chaque fonctionnalité avant de considérer le travail terminé.
Si une optimisation présente un risque de régression, la documenter et proposer une solution avant de l'appliquer.
Objectif final

Obtenir une application :

stable ;
performante ;
sans erreur HTTP 500 lors de la consultation des bulletins ;
avec une PWA fluide et réactive ;
dotée d'un système de chargement intelligent basé sur les événements réels ;
prête à évoluer vers les notifications Push et d'autres fonctionnalités futures, sans remise en cause de l'architecture existante.

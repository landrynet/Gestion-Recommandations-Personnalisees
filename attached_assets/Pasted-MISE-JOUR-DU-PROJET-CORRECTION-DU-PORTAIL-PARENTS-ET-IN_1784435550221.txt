MISE À JOUR DU PROJET – CORRECTION DU PORTAIL PARENTS ET INTÉGRATION DE LA PWA
Contexte

L'application est déjà développée et fonctionne correctement. Elle est construite avec :

Django (Backend)
HTML
CSS
Bootstrap
JavaScript
SQLite

Elle possède déjà :

le Back Office du Préfet des études ;
l'espace Enseignant ;
la gestion des élèves ;
la gestion des matières ;
la gestion des classes ;
la gestion des notes ;
la configuration des modèles de bulletins ;
la génération des bulletins ;
le portail Parents avec QR Code.

Avant toute modification, analyse complètement le projet existant afin de comprendre son architecture, ses modèles, ses vues, ses URLs, son design et sa logique métier.

Ne recrée rien depuis zéro.

Ne supprime aucune fonctionnalité existante.

Respecte entièrement l'architecture actuelle.

1. Analyse et correction du portail Parents

Le portail Parents est déjà implémenté.

Cependant, un problème existe.

Après avoir scanné le QR Code ou ouvert le portail, lorsqu'un parent saisit son code d'accès, le système retourne parfois une erreur HTTP 500 (Internal Server Error).

Je ne souhaite pas une correction basée sur des suppositions.

Je souhaite que tu :

reproduises le problème ;
consultes les logs Django sur Replit ;
identifies la cause exacte de l'erreur ;
corriges la cause réelle ;
testes entièrement le portail jusqu'à la disparition complète de cette erreur.

Vérifie notamment :

les modèles ;
les vues ;
les formulaires ;
les URLs ;
les sessions ;
les permissions ;
la récupération de l'élève via le QR Code ;
la création du code d'accès ;
le hachage et la vérification du code d'accès ;
les requêtes SQLite ;
les redirections.

Le portail ne doit jamais afficher une erreur technique à l'utilisateur.

À la place, afficher des messages clairs :

Code d'accès incorrect.
QR Code invalide.
Accès non activé.
Résultats non publiés.
Compte temporairement bloqué après plusieurs tentatives.
2. Vérification complète du parcours Parent

Tester entièrement le scénario suivant :

création d'un élève ;
génération automatique de la carte ;
génération du QR Code ;
première activation ;
création du code d'accès ;
connexion normale ;
consultation des résultats ;
publication d'une nouvelle période ;
consultation après publication ;
réinitialisation par le Préfet ;
nouvelle activation avec la même carte.

Corriger automatiquement toute anomalie détectée.

3. Intégration de la Progressive Web App (PWA)

Transformer l'application existante en Progressive Web App.

La PWA doit couvrir :

le Back Office ;
l'espace Enseignant ;
le portail Parents.

Ajouter :

manifest.json ;
Service Worker ;
icônes adaptées ;
installation sur Android ;
installation sur iPhone ;
installation sur Windows ;
installation sur macOS.

Le mode d'affichage doit être :

Standalone

L'application doit pouvoir être installée directement depuis le navigateur.

4. Cache

Mettre en cache uniquement :

CSS
JavaScript
Images
Bootstrap
Icônes
Logos
Polices

Ne jamais mettre en cache :

les notes ;
les bulletins ;
les résultats ;
les données personnelles ;
les requêtes de la base de données.

En cas d'absence d'Internet :

Afficher une belle page :

Connexion Internet indisponible

5. Préparation des notifications Push

Ne pas implémenter les notifications Push.

Préparer uniquement l'architecture afin de permettre plus tard l'intégration de Firebase Cloud Messaging.

Le code doit rester propre et facilement extensible.

6. Sécurité

Conserver la logique actuelle :

Chaque élève possède :

un QR Code unique.

Lors de la première utilisation :

le parent scanne le QR Code ;
le système détecte que le QR Code est non activé ;
le parent crée son propre code d'accès.

Lors des connexions suivantes :

scan du QR Code ;
saisie du code d'accès.

Le code d'accès doit être stocké uniquement sous forme hachée.

Ajouter une limitation des tentatives :

Après plusieurs codes incorrects :

blocage temporaire du portail.
7. Réinitialisation

Le Préfet des études peut :

Réinitialiser l'accès Parent

Lorsque cette action est exécutée :

le code d'accès actuel est supprimé ;
le QR Code redevient Non activé.

Le parent conserve exactement la même carte.

Lors de la prochaine connexion :

scan du QR Code ;
création d'un nouveau code d'accès.
8. Expérience utilisateur

Conserver exactement le design actuel.

Ne modifier ni :

la Sidebar ;
la Navbar ;
les couleurs ;
les tableaux ;
les formulaires.

Ajouter uniquement les nouveaux éléments nécessaires.

L'application doit rester fluide et responsive.

9. Tests

Avant de considérer le travail terminé :

Tester entièrement :

✓ Préfet des études

✓ Enseignant

✓ Parent

Tester également :

activation ;
connexion ;
publication ;
consultation ;
impression ;
génération des cartes ;
QR Code ;
réinitialisation.

Corriger automatiquement tout problème rencontré.

10. Rapport final

À la fin des modifications, fournir un rapport détaillé comprenant :

les fichiers modifiés ;
les nouveaux fichiers créés ;
la cause exacte de l'erreur HTTP 500 ;
les corrections effectuées ;
les tests réalisés ;
les améliorations apportées ;
les recommandations pour les prochaines évolutions.
Contraintes
Ne jamais supprimer une fonctionnalité existante.
Respecter entièrement l'architecture actuelle.
Réutiliser les composants déjà développés.
Écrire un code propre, modulaire et documenté.
Tester chaque fonctionnalité avant de passer à la suivante.
Corriger les bugs détectés au fur et à mesure.
Si une modification risque d'introduire une régression, créer une branche Git ou un point de restauration avant de l'appliquer.

Objectif final : obtenir une application stable, installable comme une PWA, sécurisée, sans erreur HTTP 500 sur le portail Parents, et prête à accueillir ultérieurement les notifications Push et d'autres fonctionnalités avancées.

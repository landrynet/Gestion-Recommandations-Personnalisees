# Phase : Refonte du module de gestion des cartes d'élèves

## Objectif

Améliorer complètement le module de génération des cartes d'élèves afin d'offrir une expérience plus moderne, plus intuitive et plus flexible. Le système doit permettre aux établissements de personnaliser facilement leurs cartes sans modifier le code de l'application.

Cette évolution ne doit pas modifier les fonctionnalités existantes de génération des cartes, mais améliorer leur présentation, leur ergonomie et leur capacité de personnalisation.

---

# 1. Nouvelle interface d'affichage

Actuellement, le système affiche simultanément le recto et le verso de chaque carte.

Cette présentation doit être remplacée par une interface plus propre.

Chaque élève doit être représenté par **une seule carte visible à l'écran**.

Par défaut, seule la **face recto** est affichée.

Le verso ne doit plus apparaître automatiquement.

---

# 2. Bascule Recto / Verso

Chaque carte doit proposer un bouton ou une action permettant de basculer entre les deux faces.

Exemples :

* Voir le verso
* Retour au recto

ou

Une animation de retournement ("flip card") fluide.

L'utilisateur doit pouvoir consulter les deux faces sans quitter la page.

La transition doit être élégante et rapide.

---

# 3. Amélioration du design

Moderniser entièrement le rendu visuel des cartes.

Le design doit être :

* professionnel ;
* élégant ;
* épuré ;
* responsive ;
* adapté à l'impression.

Les cartes doivent utiliser :

* de meilleurs espacements ;
* une hiérarchie visuelle claire ;
* des icônes discrètes ;
* une meilleure typographie ;
* des couleurs harmonieuses.

Le rendu doit rappeler les cartes professionnelles utilisées dans les établissements scolaires.

---

# 4. Gestion des modèles

Créer un véritable système de modèles de cartes.

Le préfet doit pouvoir choisir parmi plusieurs modèles prédéfinis.

Exemples :

* Modèle Classique
* Modèle Moderne
* Modèle Institutionnel
* Modèle Minimaliste
* Modèle Premium

Chaque modèle possède :

* sa disposition ;
* ses couleurs ;
* ses bordures ;
* ses polices ;
* ses emplacements des informations.

Le changement de modèle doit être instantané.

Aucune modification du code ne doit être nécessaire.

---

# 5. Personnalisation de l'établissement

Créer une section permettant au préfet de personnaliser les cartes de son école.

Il doit pouvoir modifier :

* le logo ;
* le nom de l'établissement ;
* les couleurs principales ;
* les couleurs secondaires ;
* la devise ;
* le cachet (optionnel) ;
* la signature du préfet (optionnelle) ;
* la signature du directeur (optionnelle).

Toutes les nouvelles cartes utiliseront automatiquement cette configuration.

---

# 6. Aperçu en temps réel

Ajouter un aperçu dynamique.

Lorsque le préfet modifie :

* le modèle ;
* les couleurs ;
* le logo ;
* les signatures ;

la carte affichée doit se mettre à jour immédiatement, sans recharger la page.

---

# 7. Impression

Optimiser l'impression.

Le système doit permettre :

* d'imprimer une seule carte ;
* plusieurs cartes ;
* toute une classe.

Le rendu PDF doit être parfaitement aligné.

Les dimensions doivent respecter le format standard d'une carte scolaire.

---

# 8. Gestion des informations

Le système doit permettre d'activer ou de masquer certains éléments.

Exemples :

* photo ;
* matricule ;
* classe ;
* année scolaire ;
* QR Code ;
* code-barres ;
* date de naissance ;
* sexe ;
* téléphone de l'établissement.

Chaque école pourra choisir les informations qu'elle souhaite afficher.

---

# 9. QR Code

Le QR Code doit rester intégré.

Prévoir une architecture permettant ultérieurement :

* la vérification d'authenticité ;
* la consultation sécurisée de certaines informations ;
* la validation d'une carte.

---

# 10. Gestion des modèles personnalisés

Préparer une architecture permettant, dans une phase future, de créer de nouveaux modèles sans modifier le cœur de l'application.

Les modèles devront être facilement ajoutables.

---

# 11. Responsive

Les cartes doivent être parfaitement adaptées :

* aux ordinateurs ;
* aux tablettes ;
* aux smartphones.

Le retournement Recto / Verso doit fonctionner correctement sur les écrans tactiles.

---

# 12. Performance

L'affichage des cartes doit rester fluide, même lorsqu'une classe comporte plusieurs centaines d'élèves.

Le système doit éviter les rechargements inutiles et optimiser les ressources utilisées.

---

# Résultat attendu

À la fin de cette évolution, le module des cartes d'élèves devra offrir une interface moderne, professionnelle et intuitive. Chaque élève sera représenté par une seule carte interactive pouvant être retournée pour afficher le verso. Le préfet pourra choisir parmi plusieurs modèles prédéfinis et personnaliser l'identité visuelle de son établissement (logo, couleurs, signatures, devise, etc.) sans modifier le code. L'impression restera de haute qualité et l'architecture sera prête à accueillir de futurs modèles personnalisés. Le module sera ainsi plus esthétique, plus flexible et mieux adapté aux besoins des différents établissements scolaires.

# Évolution du système : Gestion des semestres, périodes, examens, repêchage et historique enseignant

Nous souhaitons faire évoluer le fonctionnement pédagogique du système afin qu'il corresponde à l'organisation réelle de notre établissement scolaire. Cette évolution doit être intégrée sans casser les fonctionnalités existantes et en conservant toute la logique actuelle de calcul des notes, des bulletins et des résultats.

L'objectif est de simplifier le travail des enseignants, de renforcer la sécurité des données et de préparer le système à la gestion des nouvelles années scolaires.

## 1. Nouvelle organisation pédagogique

L'année scolaire sera désormais organisée en **deux semestres**.

Chaque semestre possède son propre cycle d'évaluation.

### Premier semestre

* Première période
* Deuxième période
* Examen du Premier semestre

### Deuxième semestre

* Troisième période
* Quatrième période
* Examen du Deuxième semestre
* Session de repêchage (facultative)

Le repêchage ne doit pas être actif par défaut.

Le préfet pourra décider de l'activer uniquement si cela est nécessaire.

Cette structure devra rester configurable afin de permettre des adaptations futures sans modification du code.

---

# 2. Gestion par le préfet

Le préfet est le seul responsable du cycle pédagogique.

Il pourra :

* créer les semestres ;
* activer un semestre ;
* clôturer un semestre ;
* publier les résultats d'un semestre ;
* activer ou désactiver la session de repêchage ;
* ouvrir le semestre suivant.

À un instant donné, **un seul semestre peut être actif**.

---

# 3. Interface des enseignants

Lorsqu'un enseignant se connecte, il ne doit plus voir toute l'année scolaire.

Il doit uniquement voir le semestre actuellement actif.

Par exemple :

Si le Premier semestre est actif :

L'enseignant voit uniquement :

* Première période
* Deuxième période
* Examen du Premier semestre

Il ne voit pas :

* Troisième période
* Quatrième période
* Examen du Deuxième semestre
* Repêchage

Ces éléments restent complètement masqués.

L'interface devient ainsi beaucoup plus simple et évite toute erreur.

---

# 4. Publication des résultats

Lorsque toutes les notes sont terminées :

Le préfet :

* vérifie les données ;
* calcule les résultats ;
* génère les bulletins ;
* publie officiellement les résultats.

Après publication :

Le semestre devient automatiquement verrouillé.

Les enseignants ne peuvent plus :

* modifier les notes ;
* supprimer des évaluations ;
* recalculer les résultats.

Toutes les données passent automatiquement en lecture seule.

---

# 5. Historique enseignant

Après publication, le semestre ne doit pas disparaître.

Créer un nouvel espace intitulé :

**Historique**

Cet espace permettra à chaque enseignant de consulter ses anciens semestres.

Par exemple :

Année scolaire 2026-2027

Premier semestre

* Première période
* Deuxième période
* Examen Premier semestre
* Moyennes
* Résultats
* Bulletins publiés

Toutes ces informations sont consultables uniquement.

Aucune modification ne sera possible.

Lorsque le Deuxième semestre est activé :

L'écran principal affichera uniquement le Deuxième semestre.

Le Premier semestre sera automatiquement déplacé dans l'Historique.

Ainsi, l'enseignant travaille toujours uniquement sur le semestre actif tout en gardant l'accès à ses anciens travaux.

---

# 6. Historique lié à l'enseignant

Chaque enseignant doit uniquement consulter son propre historique.

Exemple :

Un enseignant de Mathématiques ne verra que :

* ses classes ;
* ses matières ;
* ses notes ;
* ses bulletins.

Il ne pourra jamais consulter les données d'un autre enseignant.

Toutes les permissions existantes doivent être respectées.

---

# 7. Activation du Deuxième semestre

Une fois le Premier semestre publié :

Le préfet active le Deuxième semestre.

Les enseignants voient automatiquement :

* Troisième période
* Quatrième période
* Examen du Deuxième semestre

Le Premier semestre reste disponible uniquement dans l'Historique.

---

# 8. Session de repêchage

Si le préfet décide d'organiser un repêchage :

Il active la session.

Les enseignants concernés peuvent uniquement saisir les notes de repêchage.

Une fois le repêchage terminé :

* les résultats sont recalculés selon les règles de l'établissement ;
* les bulletins sont mis à jour si nécessaire ;
* la session de repêchage est automatiquement archivée.

Si aucun repêchage n'est organisé, cette fonctionnalité reste totalement invisible.

---

# 9. Portail Parent

Le portail parent ne doit afficher que les résultats officiellement publiés.

Par exemple :

Lorsque seul le Premier semestre est publié :

Le parent voit uniquement :

* Bulletin du Premier semestre
* Résultats du Premier semestre

Le Deuxième semestre reste invisible.

Après publication du Deuxième semestre :

Le portail affiche :

* Bulletin annuel
* Résultat final
* Décision finale

Toutes les données des semestres précédents restent consultables dans l'Historique du portail parent.

---

# 10. Nouvelle année scolaire

Lorsqu'une nouvelle année scolaire est activée :

Le système recommence automatiquement le même cycle.

Par défaut :

Premier semestre

* Première période
* Deuxième période
* Examen du Premier semestre

Deuxième semestre

* Troisième période
* Quatrième période
* Examen du Deuxième semestre
* Repêchage (désactivé par défaut)

Le système doit être capable de réutiliser cette structure automatiquement lors de chaque nouvelle année scolaire.

---

# 11. Archivage

À la clôture d'une année scolaire :

Les deux semestres deviennent définitivement archivés.

Ils restent consultables :

* par le préfet ;
* par les enseignants (uniquement leurs propres données) ;
* par les parents (uniquement les données de leurs enfants).

Aucune modification ne sera possible.

---

# 12. Sécurité

Le système doit empêcher toute modification :

* d'un semestre publié ;
* d'un semestre archivé ;
* d'une année scolaire clôturée.

Toutes les données archivées deviennent définitivement en lecture seule.

---

# 13. Compatibilité avec les bulletins

Cette nouvelle organisation ne doit pas casser le fonctionnement actuel des bulletins.

Les bulletins devront continuer à récupérer les données officielles enregistrées dans chaque semestre.

Le portail parent devra continuer à utiliser les données des bulletins comme source officielle.

Aucun recalcul ne devra être effectué dans le portail parent.

---

# 14. Évolutivité

L'architecture doit être suffisamment flexible pour permettre à l'avenir :

* un nombre différent de semestres ;
* un nombre différent de périodes ;
* plusieurs sessions d'examens ;
* plusieurs sessions de repêchage ;
* des règlements scolaires différents.

Ces évolutions devront pouvoir être réalisées sans refonte majeure du système.

---

# Objectif

L'objectif est de mettre en place une gestion pédagogique **professionnelle, simple, sécurisée et évolutive**. Les enseignants travailleront uniquement sur le semestre actif afin de réduire les erreurs et de simplifier leur interface. Une fois les résultats publiés, les données seront automatiquement verrouillées et déplacées dans un espace **Historique** consultable en lecture seule. Le préfet conservera le contrôle complet du cycle de vie des semestres (activation, publication, archivage et ouverture du semestre suivant), tandis que les parents ne consulteront que les résultats officiellement publiés. Cette architecture devra être parfaitement compatible avec la gestion des années scolaires, les bulletins, le portail parent et les futures évolutions du système, tout en garantissant la cohérence, la sécurité et la conservation intégrale des données historiques.
